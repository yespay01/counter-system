# 1-3. Solapi 운영 조사

> 목적: Solapi를 고정 공급자로 전제하고, 실제 운영에 필요한 준비사항과 구현 보완 포인트를 정리
> 주의: 이 문서는 회의 전 리서치 원본이다. 공식 기준은 `2026-03-17 1-3 Solapi 운영 조사 회의` 결과를 우선한다.

## 회의 이후 공식 결정

- 공급자는 `Solapi` 유지
- 기본 메시지 정책은 `알림톡 우선, 실패 시 자동 SMS fallback (기본 ON)`
- 채널 구조는 `플랫폼 공유 채널(기본) + 테넌트 개별 채널(선택)` 하이브리드
- 브라우저 직접 발송은 `MVP 이전 폐기`
- Solo MVP `admin/messages`에는 `PFID / 발신번호 / 템플릿 식별자 입력 + 발송 로그 확인`이 포함
- Solapi 운영 준비는 `Phase 2~3부터 병행 시작`

아래 본문은 리서치 시점의 판단 근거와 운영 메모로 본다.

## 조사 범위

- Solapi를 계속 사용하는 전제
- 대안 비교는 생략
- 알림톡/SMS 운영 준비, 발송 단가, 템플릿 정책, 현재 코드의 보완 필요사항 정리

## 1. 리서치 기준 결론 요약

- 공급자는 `Solapi`로 유지
- 기본 채널은 `알림톡(ATA)` 우선, 실패 시 `SMS` 대체 발송 검토
- 발송 로직은 `브라우저 직접 호출`이 아니라 `서버 API`로 옮겨야 함
- 템플릿, PFID, 발신번호, 발송 로그, 실패 재시도는 운영 설계에 포함해야 함

해석: 이미 제품 방향과 기존 구현이 Solapi에 맞춰져 있으므로 공급자를 바꿀 이유는 약하다. 대신 `운영 안정성`과 `보안`을 위해 구조를 바꿔야 한다.

## 2. 운영 준비사항

### 문자(SMS/LMS) 발송
- Solapi 가이드 기준 문자 발송 전 `발신번호 등록`이 필요하다.
- 발신번호 등록은 본인 인증 또는 증빙서류 제출이 필요할 수 있다.
- 서류 검토는 보통 `1~3 영업일`이 걸릴 수 있다.

### 카카오 알림톡
- Solapi 가이드 기준 `카카오 비즈니스 채널`이 먼저 필요하다.
- Solapi에 채널을 연동하면 채널별 `PFID`가 발급된다.
- 알림톡은 `등록된 템플릿`으로만 보낼 수 있고, 템플릿 검수는 카카오가 진행한다.
- 템플릿 검수도 보통 `1~3 영업일`이 걸린다.

### 실무 체크리스트
1. 사업자 계정 개설
2. 발신번호 등록
3. 카카오 비즈니스 채널 개설
4. Solapi에 채널 연동 후 PFID 확보
5. `발급`, `호출예정`, `호출`, `취소/지연` 템플릿 등록
6. 운영용 잔액/자동충전 설정

## 3. 가격 기준

2026-03-16 기준 Solapi 공개 가격:

- 알림톡 `13원/건`
- SMS `18원/건`
- LMS `45원/건`
- 월 기본료 및 API 사용료 `무료`

### 해석
- 대기 시스템은 광고성보다 정보성 메시지가 많으므로 기본 채널은 `알림톡`이 맞다.
- 알림톡 실패 시 SMS 대체 발송을 붙이면 전달률은 좋아지지만 원가가 바로 올라간다.
- 따라서 상품은 `월 포함량 + 초과 과금`이 가장 단순하다.

## 4. 알림톡 운영 정책 포인트

- 알림톡은 정보성 메시지 전용이다.
- 2025-07-01 공지 기준 쿠폰/포인트 등 영리 목적 문구 모니터링이 강화됐다.
- 카카오 친구톡은 Solapi 공지 기준 `2025-12-31` 종료 예정이다.

### 제품 적용 해석
- 현재 서비스는 `순번 발급`, `호출`, `호출예정` 알림 중심이라 알림톡 정책과 잘 맞는다.
- 마케팅성 문구, 할인 유도 문구, 과장된 표현은 템플릿에서 배제해야 한다.
- 친구톡 의존 설계는 피하고, 필요하면 `SMS` 또는 향후 `브랜드 메시지`를 별도 검토한다.

## 5. 현재 코드 기준 보완 필요사항

[`kiosk-sms/index.html`](/mnt/c/Users/sakai/OneDrive/바탕 화면/counter-system/kiosk-sms/index.html) 기준:

- API Key/Secret을 브라우저 설정 패널과 `localStorage`에 저장하고 있다.
- 브라우저에서 Solapi API를 직접 호출하고 있다.
- `generateSignature()`는 실제 HMAC 서명이 아니라 임시 랜덤 문자열을 반환한다.
- `pfId`가 `'your_pfId'` placeholder 상태다.
- 실패 시 서버 로그, 재시도, 대체발송, 상태 조회가 없다.

### 결론
- 현재 구현은 `프로토타입/테스트용`으로는 의미가 있지만, 운영 구조로는 부적합하다.
- SaaS 전환 시 아래처럼 바꿔야 한다.
  - 키/시크릿은 서버 환경변수 저장
  - 키오스크는 `/api/public/:slug/queue/issue`만 호출
  - 서버가 Solapi 발송 수행
  - 결과를 `sms_logs`에 저장
  - 실패 시 상태코드 저장 및 선택적 SMS fallback

## 6. 권장 발송 플로우

1. 키오스크가 번호 발급 요청
2. 서버가 대기열 저장
3. 서버가 알림톡 발송 시도
4. 성공 시 `sent`
5. 실패 시:
   - 정책상 허용되면 SMS 대체 발송
   - 그래도 실패하면 `failed` 로그 저장
6. 관리자 화면에 `잔여량`, `실패건수`, `최근 오류코드` 표시

## 7. 상태 코드 / 운영 로그

- Solapi 메시지 상태 코드는 4자리 숫자 기준으로 관리된다.
- `2000`은 정상 접수, `3000`은 이통사 접수 완료, `4000`은 발송 완료다.
- `1xxx`, `2xxx`, `3xxx` 계열은 각 단계 오류를 뜻한다.

### 운영 적용
- 사용자 화면에는 상태코드 원문을 노출하지 말고
  - `접수됨`
  - `발송중`
  - `완료`
  - `실패`
  로 단순화한다.
- 운영자 화면에는 원본 상태코드와 실패 원인을 남겨야 한다.

## 8. 리서치 기준 권장안

- Solapi는 유지
- 알림톡 우선 + 선택적 SMS fallback
- 메시지 발송은 반드시 서버로 이관
- 운영 필수 항목:
  - 발신번호 관리
  - PFID 관리
  - 템플릿 버전 관리
  - 발송 로그
  - 실패 재시도 정책
  - 잔액/자동충전 모니터링

### 공식 결정과 다른 부분
- 이 문서는 `선택적 SMS fallback`을 전제로 적었지만, 현재 공식 기준은 `자동 SMS fallback 기본 ON`이다.
- 이 문서는 운영 옵션 관점의 메모이며, 실제 MVP 범위는 회의 결과의 `admin/messages`와 상위 계획 문서를 따른다.
