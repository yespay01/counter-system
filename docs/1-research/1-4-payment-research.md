# 1-4. 결제 서비스 조사

> 상태: ✅ 1차 완료 (2026-03-16, Codex)
> 목적: SaaS 구독 결제 연동 방식을 결정하고, 초기 도입 결제사를 선정
> 주의: 이 문서는 회의 전 리서치 원본이다. 공식 기준은 `2026-03-17 1-4 결제 서비스 조사 회의` 결과를 우선한다.

---

## 회의 이후 공식 결정

- Solo MVP 결제 방식은 `계좌이체 수동 처리 (PG 미연동)`
- PG사는 현재 `미정`
- PG 재검토 기준은 `업소 50개 이상`
- 설계는 `BillingService` 경계 + `ManualBillingService` 구현체 기준
- 구독 상태 흐름은 `trial -> active -> past_due -> suspended`
- sandbox → 유료 전환은 운영자가 계좌이체 확인 후 수동으로 `active + sandbox_mode=false`
- Solo MVP에는 수동 결제 상태 관리를 위한 최소 `/superadmin` 화면이 포함

아래 본문은 회의 전 리서치 판단 근거와 비교 자료로 유지한다.

## 조사 범위

- 월 구독형 SaaS에 맞는 정기결제 수단 확인
- `토스페이먼츠`와 `포트원` 비교
- 초기 출시 기준 권장안 도출

## 1. 리서치 기준 결론 요약

- 초기 결제는 `토스페이먼츠 직접 연동` 권장
- 이유: 구독형 SaaS에 필요한 카드 자동결제를 공식 지원하고, 구조가 단순하며 가격도 공개돼 있다
- `포트원`은 다중 PG, 슈퍼빌링키, 재시도/재예약 운영 기능이 장점이지만 현재 단계에서는 과하다

해석: 이 제품은 지금 `결제 인프라 사업`이 아니라 `현장형 순번대기 SaaS`를 빨리 검증해야 한다. 초기에는 결제 추상화보다 단순한 직접 연동이 낫다.

## 2. 토스페이먼츠

### 장점
- 공식 문서 기준 자동결제(빌링)를 지원한다.
- `authKey → billingKey 발급 → billingKey로 월별 승인` 흐름이 명확하다.
- 2026-03-16 기준 공개 가격 페이지가 있다.
- 정산주기, 가입비, 연관리비, 일반 카드 수수료가 공개되어 있다.

### 확인된 사항
- 자동결제는 `리스크 검토 및 추가 계약` 후 사용할 수 있다.
- 정기 구독형 서비스가 아니라면 정책적으로 제한된다.
- 카드뿐 아니라 계좌 자동결제도 지원한다.
- 2026년 1월 릴리즈 노트 기준 `퀵계좌이체 자동결제(빌링)`가 새로 추가됐다.

### 공개 가격
- 가입비 `220,000원`
- 연관리비 `110,000원`
- 신용/체크카드 일반 `3.4%`
- 계좌이체 `2.0%`
- 브랜드페이 `4.3%`

### SaaS 적합성 해석
- 우리 서비스는 전형적인 월 구독형이라 정책 적합성이 높다.
- 초기에는 `카드 자동결제`만 붙여도 충분하다.
- 나중에 저수수료가 중요해지면 `계좌 자동결제`를 검토할 수 있다.

## 3. 포트원

### 장점
- V2 API 기준 빌링키 발급, 삭제, 조회, 예약결제를 지원한다.
- 여러 PG를 하나의 API/SDK로 추상화할 수 있다.
- 공식 도움말 기준 `슈퍼빌링키`, 실패 건 `재예약/재시도`, `스마트라우팅`, 통합 결제 관리 기능을 제공한다.
- PG 수수료 비교표가 공개되어 있어 계약 판단이 쉽다.

### 확인된 사항
- PortOne V2는 `Authorization: PortOne MY_API_SECRET` 형식 인증을 사용한다.
- `Idempotency-Key`로 멱등 재시도를 지원한다.
- PG별 빌링/수기 수수료 비교표가 공개되어 있다.
- 포트원 자체 부가서비스 요금은 일부가 상담형 또는 변동형이다.

### 공개 가격/비용 성격
- 포트원 헬프센터 공개표 기준 토스페이먼츠의 `빌링/수기` 수수료는 `3.23%`, NHN KCP는 `3.20%`, KG이니시스는 `3.20%`다.
- 다만 포트원 자체 요금은 서비스 조합에 따라 달라질 수 있고, 일부는 별도 문의가 필요하다.

### SaaS 적합성 해석
- 다중 PG 운영, 장애시 우회, 재시도 자동화가 중요해지면 가치가 커진다.
- 하지만 초기 유료 고객이 몇 곳 안 되는 단계에서 넣으면 설계와 계약 복잡도만 커질 가능성이 높다.

## 4. 비교

### 토스페이먼츠가 더 나은 점
- 구독형 SaaS에 바로 필요한 기능만 직접 구현 가능
- 문서와 가격이 명확
- 계약 상대가 단순
- 운영 복잡도 낮음

### 포트원이 더 나은 점
- 여러 PG를 한 번에 다루기 쉬움
- 실패 재시도/재예약, 슈퍼빌링키 등 운영 기능이 풍부
- 장기적으로 결제 인프라를 고도화하기 좋음

## 5. 리서치 기준 권장안

### Phase 1
- `토스페이먼츠 직접 연동`
- 결제 수단: `카드 자동결제`
- 청구 주기: `월 단위`
- 서버가 billingKey를 저장하고, 매월 스케줄러로 자동결제 승인

### Phase 2 조건
아래 중 2개 이상 충족 시 포트원 재검토:
- 월 거래액이 의미 있게 증가
- 결제 실패 재시도 자동화가 중요
- 여러 PG 비교/분산이 필요
- 해외 결제 또는 다양한 수단 확장 필요

## 6. 구현 메모

- 결제 provider 인터페이스는 처음부터 분리한다.
- 하지만 실제 구현체는 `TossPaymentsBillingService` 하나만 둔다.
- 구독 테이블에는 아래가 필요하다.
  - `billing_provider`
  - `billing_customer_key`
  - `billing_key`
  - `plan`
  - `status`
  - `next_billing_at`
  - `last_billed_at`
  - `last_failed_at`

### 공식 결정과 다른 부분
- 이 문서는 `토스페이먼츠 직접 연동 + 카드 자동결제`를 권장했지만, 현재 공식 기준은 `PG 미연동 + 계좌이체 수동 처리`다.
- `billingKey`, `TossPaymentsBillingService`, 자동 청구 스케줄러는 현재 구현 범위가 아니라 회의 전 비교안의 일부다.
- 다만 `BillingService` 경계 분리, 구독 상태 컬럼 준비 같은 구조적 제안은 여전히 참고 가능하다.
