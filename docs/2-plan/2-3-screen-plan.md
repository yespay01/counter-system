# 2-3. 화면 설계

> 상태: 🟢 v2 (2026-03-17, 1-1/1-2 회의 결과 반영)
> 목적: 현재 HTML 기반 운영 화면을 SaaS 구조에 맞게 재배치하고, 새로 필요한 관리자/가입 화면 범위를 정의
> **Solo MVP 우선**: 태블릿 1대 단일 화면(번호발급+호출 통합)을 먼저 완성한다

---

## 1. 설계 원칙

- **Solo MVP 우선**: 태블릿 1대로 사장 혼자 운영하는 `/:slug/solo` 단일 화면을 먼저 완성
- 현장 운영 화면은 `큰 터치 영역`, `짧은 동선`, `즉시 반응`이 우선
- 관리자 화면은 `설정/로그/구독` 중심
- 현재 HTML의 강점인 `직관성`은 유지하고, 복잡한 기능은 뒤로 미룬다
- 데스크톱/태블릿 공용 대응이 필요하다
- 샌드박스 모드에서는 모든 화면에 배너 표시 + 메시지 발송 시 미리보기 팝업

## 2. 전체 화면 목록

### Solo MVP 화면 (1차 출시)
- `/:slug/solo` — 키오스크 + 호출 통합 단일 화면 (태블릿 1대)
- `/:slug/join` — QR 진입용 공개 접수 화면
- `/:slug/admin/dashboard` — 관리자 대시보드
- `/:slug/admin/settings` — 업체/카테고리 설정
- `/:slug/admin/messages` — 메시지 로그/설정 (PFID 입력 포함)
- `/:slug/admin/subscription` — 구독/결제 상태 (기본)
- `/superadmin` — 수동 결제 상태 관리용 최소 운영 화면
- `회원가입`
- `로그인`

### Standard 화면 (2차 확장)
- `/:slug/kiosk` — 전용 키오스크 (다창구 환경)
- `/:slug/kiosk-sms` — SMS형 전용 키오스크
- `/:slug/counter` — 창구 직원 화면
- `/:slug/display` — 대기실 TV 디스플레이
- `랜딩`

### 관리자 화면
- `/:slug/admin/dashboard`
- `/:slug/admin/settings`
- `/:slug/admin/categories`
- `/:slug/admin/messages`
- `/:slug/admin/subscription`

### 운영자 화면
- `/superadmin` (Solo MVP 최소 포함)

## 3. 사용자 역할별 동선

### 고객
1. 태블릿 또는 QR로 접수 화면 접근
2. 카테고리 선택
3. 필요 시 전화번호 입력
4. 음식점형 업종은 필요 시 메뉴/주문 항목 선택
5. 번호 발급 결과 확인

### 직원
1. 창구 화면 로그인 또는 기기 진입
2. 창구 번호 선택
3. 다음 호출 / 재호출 / 완료
4. 현재 창구 상태 확인

### 관리자
1. 로그인
2. 업체 설정 입력
3. 카테고리/카운터 설정
4. 메시지 템플릿/PFID 연결
5. 구독/사용량/로그 확인

## 4. 화면별 설계

## 4-0. Solo 통합 화면 `/:slug/solo` ⭐ Solo MVP 핵심

### 목적
- 사장 혼자 태블릿 1대로 번호 발급 + 호출 + 완료를 모두 처리
- 고객은 화면을 보고 카테고리 선택 → 사장이 같은 화면에서 호출

### 구성
- **상단 바**: 업체명, 현재 시각, 연결 상태, 샌드박스 배너(테스트 모드 시)
- **좌측 (고객 영역)**: 카테고리 선택 버튼 (대형 터치 영역)
  - 기본 모드: 카테고리 선택 후 즉시 발급
  - 전화번호 모드: 전화번호 입력 → 카테고리 선택 → 번호 발급
  - 음식점형 업종: 전화번호 입력 → 메뉴/주문 항목 선택 → 번호 발급
  - 번호 발급 완료 시 성공 모달 (번호 + 예상 대기)
  - QR 접수 안내 배지 / QR 코드 노출 가능
- **우측 (사장 영역)**: 현재 대기열 목록 + 호출 버튼
  - 다음 호출 / 재호출 / 완료 버튼
  - 현재 응대 중 번호 표시

### UX 원칙
- 고객 영역과 사장 영역이 한 화면에 있되 명확히 구분
- 호출 버튼은 크고 즉시 반응
- 메시지 발송 시 샌드박스 팝업 표시 (테스트 모드)
- 전화번호 입력과 메뉴 선택은 터치 한두 번 안에 끝나야 한다
- QR 접수는 태블릿 혼잡을 줄이는 보조 동선으로 동작해야 한다

### 비고
- Standard 전환 시 고객 전용 kiosk와 사장 전용 counter로 분리

## 4-0A. 공개 접수 화면 `/:slug/join`

### 목적
- 매장 QR을 스캔한 고객이 자기 휴대전화에서 직접 접수
- 태블릿 앞 대기열이 길 때 셀프 입력 동선 제공

### 구성
- 업체명 / 기본 안내
- 전화번호 입력
- 카테고리 선택
- 음식점형 업종용 메뉴/주문 항목 선택
- 발급 결과 화면: 대기번호, 안내 문구, 알림톡 발송 상태

### UX 원칙
- 모바일 세로 화면 우선
- 전화번호 입력과 선택 항목 수를 최소화
- 발송 실패 여부와 무관하게 번호 발급 결과는 즉시 보여준다

## 4-1. 랜딩

### 목적
- 제품 설명
- 타겟 업종 제시
- 무료 체험 유도

### 핵심 영역
- 헤더: 서비스명, 로그인, 무료 체험 시작
- 히어로: “호떡집부터 학원까지 — 태블릿 1대로 순번 관리”
- 문제/해결 섹션
- 주요 기능 3개
- 가격 요약 (기본 구독료 + 메시지 건당 과금)
- 하단 CTA (샌드박스 무료 체험)

### 비고
- 1차 타겟: 소규모 현장 판매 업종 (호떡집, 분식 포장점, 팝업스토어, 소형 베이커리)
- 외식업 웨이팅 앱처럼 보이지 않게 문구 조정
- 랜딩은 Standard 출시 전에 구현해도 무방 (Solo MVP에는 불포함)

- 비밀번호

### 후속 흐름
- 가입 완료 후 온보딩 시작
- trial 상태 자동 생성

## 4-3. 로그인

### 입력 항목

### 로그인 후 이동
- 기본: `/:slug/admin/dashboard`

## 4-4. 키오스크 일반형 `/:slug/kiosk`

### 목적
- 전화번호 없이 번호표 발급

### 구성
- 상단: 로고 / 업체명 / 현재시간
- 중앙: 카테고리 버튼 그리드
- 하단: 안내문
- 숨김 설정 패널: 프린터 설정, 시스템 상태

### UX 원칙
- 버튼은 대형 터치 영역
- 클릭 후 즉시 성공 피드백
- 프린터 오류가 있어도 운영 흐름은 멈추지 않게 처리

## 4-5. 키오스크 SMS형 `/:slug/kiosk-sms`

### 목적
- 전화번호 입력 후 번호 발급 + 알림톡/SMS

### 구성
- 상단: 업체명 / 시간 / 기본 안내
- 중앙 상단: 전화번호 입력 영역
- 중앙 중단: 메뉴/주문 항목 선택 영역 (음식점형 업종 선택 기능)
- 중앙 하단: 카테고리 버튼
- 성공 모달: 번호, 전화번호, 예상대기 안내
- 숨김 설정 패널: 운영 상태 표시

### UX 원칙
- 숫자 입력이 가장 먼저 눈에 들어와야 함
- 휴대전화 유효성 검사 즉시 피드백
- 메시지 실패 시에도 번호 발급은 성공 처리

## 4-6. 창구 화면 `/:slug/counter`

### 목적
- 직원이 가장 빠르게 호출할 수 있어야 함

### 구성
- 상단: 업체명, 창구 선택, 연결 상태
- 메인 카드: 현재 응대 번호
- 액션 버튼: 다음 호출 / 재호출 / 완료 / 직접 호출
- 보조 패널: 현재 대기열, 창구 담당 카테고리, 최근 호출

### UX 원칙
- 메인 CTA는 `다음 호출`
- 재호출과 완료는 같은 위치 패턴 유지
- 실시간 연결 이상 시 명확한 경고 표시

## 4-7. 디스플레이 `/:slug/display`

### 목적
- 멀리서도 읽히는 호출 안내

### 구성
- 좌측: 현재 호출 번호, 창구 번호
- 우측 상단: 대기중 번호 목록
- 우측 하단: 창구별 상태
- 관리자 숨김 진입: 오디오, 테스트, 디버그

### UX 원칙
- 번호가 최우선
- 애니메이션은 호출 순간만 강조
- 오디오 상태를 명확히 표시

## 4-8. 관리자 대시보드 `/:slug/admin/dashboard`

### 목적
- 운영 상태 한눈 확인

### 카드
- 오늘 발급 수
- 현재 대기 수
- 오늘 발송 수
- 실패 건수
- 구독 상태

### 하단 영역
- 최근 발송 실패 로그
- 구독 만료 / trial 종료 경고
- 빠른 설정 이동 버튼

## 4-9. 관리자 설정 `/:slug/admin/settings`

### 설정 항목
- 업체명
- 업종
- 운영 시간
- 카운터 수
- 번호표 자릿수/접두어 방식
- 기본 키오스크 모드
- 디스플레이 테마

## 4-10. 관리자 카테고리 관리 `/:slug/admin/categories`

### 기능
- 카테고리 추가 / 수정 / 비활성화
- 표시 순서 변경
- 카운터별 담당 카테고리 매핑

### 예시
- 학원: 중국어 / 영어 / 상담
- 의원: 초진 / 재진 / 수납
- 민원창구: 전입 / 등본 / 제증명

## 4-11. 관리자 메시지 `/:slug/admin/messages`

### 기능
- Solapi 연결 상태
- PFID 입력 / 확인
- 발신번호 입력 / 확인
- 템플릿 식별자 입력 / 관리
- 최근 발송 로그 / 실패 사유

### Solo MVP 범위
- sandbox 발송 로그 확인
- 연결 상태와 기본 발송 결과 확인
- PFID / 발신번호 / 템플릿 식별자 입력
- 채널 구조: 플랫폼 공유 채널 기본 + 테넌트 개별 채널 선택
- 메시지 정책 표시: 알림톡 우선, 실패 시 자동 SMS fallback
- 번호 발급 시 사용된 전화번호 / 진입 채널 / 메뉴 선택값 확인
- 현재 월 기준 `sent / failed / previewed` 구분과 채널별 성공 건수 확인
- 상세 정산 금액과 운영 자동화는 후속 Phase에서 확장

### 하위 탭
- 템플릿
- 발송 로그
- 사용량

## 4-12. 관리자 구독 `/:slug/admin/subscription`

### 기능
- 현재 플랜
- 다음 결제일
- 포함 메시지 / 사용량
- 초과 사용량
- 월별 성공 발송 건수
- 월별 예상 메시지 청구액
- 계좌이체 안내 / 확인 상태
- 결제 실패 상태 안내

### Solo MVP 범위
- trial / 플랜 상태 표시
- 계좌이체 안내와 기본 상태 확인
- 월별 메시지 성공 건수와 예상 청구액 확인
- 자동 청구 상세와 정산 고도화는 후속 Phase에서 확장

## 4-13. 슈퍼관리자 `/superadmin`

### 초기 범위
- 전체 업체 목록
- trial / active / past_due / suspended 상태
- 수동 결제 확인 및 상태 변경
- 최근 결제 실패 고객
- 최근 메시지 실패율 높은 고객

### 비고
- 초기에는 내부 운영용 최소 화면만 구현

## 5. 공통 컴포넌트

- 상단 상태 배지: 연결 상태, 구독 상태
- 숫자 KPI 카드
- 로그 테이블
- 확인 모달
- 토스트 메시지
- 빈 상태(empty state)

## 6. 반응형 기준

### 키오스크 / 디스플레이
- 가로형 태블릿 / 모니터 기준 우선
- 세로형 태블릿 대응 필요

### 관리자
- 데스크톱 우선
- 태블릿까지는 지원
- 모바일은 조회 중심 최소 대응

## 7. 디자인 방향

- 현재 제품의 강한 대비와 큰 타이포는 유지
- 관리자 화면은 현장 화면보다 차분하게
- 공개/현장 화면은 버튼과 숫자를 크게
- 한국어 우선 UI
