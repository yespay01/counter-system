# 2-1. 서비스 기획 확정

> 상태: ✅ 확정안 v2 (2026-03-17, Codex)
> 전제: 틈새시장 집중, Solapi 유지, 결제는 Solo MVP에서 PG 미연동 + 계좌이체 수동 처리

---

## 1. 서비스 포지셔닝

이 서비스는 예약형 레스토랑 웨이팅 앱이 아니라 `혼잡 시간대용 초경량 대기 도구`에서 출발하는 서비스다.

### 초기 타겟
1. `소규모 현장 판매 업종`
2. `포장 / 테이크아웃 중심 매장`
3. `팝업 / 행사형 운영`

### 의도적으로 제외하는 시장
- 예약형 / 좌석형 레스토랑 웨이팅 앱 정면 경쟁
- 대형 은행/대형 병원 엔터프라이즈 구축형 수주
- 복잡한 예약 플랫폼 기능

### 2차 확장 타겟
- `학원`
- `의원 / 검진센터`
- `민원창구`

## 2. 핵심 문제 정의

초기 타겟 고객은 아래 문제를 공통으로 가진다.

- 줄이 길어지는 시간대에만 잠깐 쓸 간단한 도구가 없다
- 고객을 한 줄로 세워두면 체감 불편이 크다
- 사장 혼자 운영해서 복잡한 창구 시스템을 도입할 여력이 없다
- 태블릿 1대로 번호 발급과 간단 호출만 해도 충분한 경우가 많다
- 유료 도입 전에 전체 흐름을 충분히 시험해보고 싶다

## 3. MVP 범위

### 포함
- 번호 발급
- 간단 호출 / 완료
- 태블릿 1대 운영 모드
- 전화번호 입력 후 알림톡 수신
- QR 진입 후 고객 직접 전화번호 입력
- 음식점형 업종의 메뉴/주문 항목 선택 후 번호 발급
- 알림톡 우선 발송 + 실패 시 자동 SMS fallback
- 샌드박스(테스트 모드)
- 카테고리 직접 설정
- 일일 대기열 초기화
- 관리자 기본 화면 (`admin/dashboard`, `admin/settings`, `admin/messages`, `admin/subscription`)
- 최소 `/superadmin` 운영 화면
- 기본 사용량 확인

### 제외
- 온라인 예약
- 고객용 모바일 앱
- 지점 다중 관리
- 상세 통계 대시보드
- 직원별 권한 세분화
- 외부 EMR/CRM/POS 연동
- 다창구 / 디스플레이 중심 운영은 MVP 이후 확장

## 4. 제품 구성

### Solo
- 태블릿 1대
- 번호 발급 + 간단 호출
- 태블릿에서 전화번호 입력 후 번호 발급
- QR로 공개 접수 화면 진입 후 고객 직접 전화번호 입력
- 음식점형 업종은 번호 발급 전 메뉴/주문 항목 선택 가능
- 알림톡 우선 + 실패 시 자동 SMS fallback (기본 ON)
- 플랫폼 공유 채널 기본 + 테넌트 개별 채널 선택
- PFID / 발신번호 / 템플릿 식별자 입력과 발송 로그 확인
- trial / active 상태 확인과 수동 결제 상태 관리
- 디스플레이 / 다창구 없음
- 1인 운영 매장용 MVP

### Standard
- Solo 기반 + 창구/디스플레이 확장
- 학원 / 의원 / 민원창구 대응
- 다창구 운영
- 관리자 설정과 운영 로그 강화

### Pro
- 다지점 대비 구조만 열어두고 초기 가격표에는 노출하지 않음

## 5. 가격 가안

### 추천 월 구독
- `Solo`: 추후 확정
- `Standard`: 추후 확정
- `Pro`: 상담형

### 메시지 정책
- 기본 구독료와 메시지 비용을 분리
- 실제 발송 건수만 건당 과금
- 샌드박스 테스트 모드에서는 실제 메시지 비용 없음
- 과금 대상 메시지는 `issue`, `ready`, `call` 3종으로 본다
- `sent` 성공 건만 billable로 집계하고, `previewed` / `failed`는 과금에서 제외한다
- Solapi 원가와 업소 청구 단가는 다를 수 있다
- 발송 시점의 `provider unit cost`, `billing unit price`를 스냅샷으로 남겨 월 정산 기준으로 사용한다

### 현재 확정 방향
- `기본 구독료 + 메시지 건당 별도 과금`
- 포함량 묶음 방식은 `1-1 회의` 결과 보류

해석: 사용자 결정 기준으로는 메시지 포함 플랜보다 `기본료 + 실제 사용량 과금`이 더 맞다.

## 6. 결제 정책

- Solo MVP 결제는 `계좌이체 수동 처리`
- PG사는 현재 `미정`
- 청구 주기는 `월 단위`
- `Solo`, `Standard`는 기본 구독료 결제
- 메시지는 실제 사용량 기준 별도 청구
- 메시지 사용량 청구는 `성공 발송 건수 x 발송 시점 청구 단가` 기준으로 산정한다
- 예: Solapi 알림톡 원가가 `14원`일 때 업소 청구 단가는 `18원` 또는 `20원`으로 운영 가능하다
- 수동 결제 상태 관리를 위한 최소 `/superadmin` 화면 포함

해석: 현재 단계의 핵심은 자동 청구 인프라가 아니라 현장 SaaS 검증이다. Solo MVP에서는 `BillingService` 경계와 수동 운영 흐름을 먼저 두고, PG는 업소 50개 이상 시점에 재검토한다.

## 7. 무료 체험 정책

- 무료 체험은 `샌드박스(테스트 모드)`로 재설계
- 실제 메시지 발송 없이 번호 발급, 호출, 흐름 검증 가능
- 고객은 테스트 후 구독을 결정

이유:
- 실제 영업 전에 충분히 만져볼 수 있어야 한다.
- 메시지 원가 없이 테스트 경험을 제공할 수 있다.
- 복잡한 인증/어뷰징 방지보다 테스트 모드가 초기 운영에 더 단순하다.

## 8. 설치 / 온보딩 방식

### 기본 전략
- `셀프 설치형` 우선
- 필요 시 유상 원격 설치 지원

### 온보딩 흐름
1. 회원가입
2. 업체명 / slug / 카테고리 입력
3. 샌드박스 모드에서 번호 발급 / 호출 / 전화번호 입력 / QR 접수 테스트
4. 구독 결정
5. 계좌이체 안내 및 입금
6. 운영자 입금 확인 + 활성화
7. Solapi 발신번호/PFID/템플릿 식별자 연결
8. 실사용 시작

## 9. 운영 정책

- 발신 실패가 있어도 번호 발급은 유지
- 메시지는 보조 수단이고, 기본 운영은 화면 호출 기준
- 관리자에게 잔여 건수와 실패 로그를 항상 보여줌
- 관리자에게 월별 성공 발송 건수와 예상 메시지 청구 금액을 보여줌
- 대기 데이터는 일별 세션 단위로 보관
- MVP는 `Solo 우선`, 다창구/디스플레이는 확장 모드로 분리
- 번호 발급은 `즉시 발급`, `전화번호 입력 후 발급`, `QR 셀프 접수` 3가지 흐름을 허용한다
- 음식점형 업종은 번호 발급 전 `메뉴/주문 항목 선택`을 추가로 받을 수 있다
- 메시지 정책 기본값은 `알림톡 우선, 실패 시 자동 SMS fallback`
- 채널 구조는 `플랫폼 공유 채널(기본) + 테넌트 개별 채널(선택)` 하이브리드
- `admin/messages`에서 PFID / 발신번호 / 템플릿 식별자 입력과 발송 로그를 다룬다
- `admin/messages`와 `admin/subscription`에서 월별 billable 건수, 채널별 집계, 예상 청구액을 확인할 수 있어야 한다
- Solo MVP 구독 전환은 운영자가 계좌이체 확인 후 수동으로 `trial → active` 처리
- PG 연동은 `업소 50개 이상` 시점을 공식 재검토 기준으로 둔다
- `/superadmin`에서 수동 결제 상태를 관리한다
- 민감정보는 최소 저장:
  - 전화번호
  - 발송 결과
  - 대기 로그

## 10. 성공 기준

### 출시 1차 목표
- 유료 고객 `3~5곳`
- 월 발송 실패율 `1% 이하`
- 셀프 설치 완료율 `70% 이상`
- `소규모 현장 판매 업종`에서 반복 가능한 온보딩 확보

### 제품 적합성 판단 기준
- 고객이 `종이번호표보다 낫다`고 바로 체감하는가
- 설치가 `30분 안`에 끝나는가
- 직원 교육 없이 호출 흐름이 이해되는가
