# 안건 — 1-2 기술 리서치 (2026-03-16 18:26)

> 작성: Codex
> 요청자: 사용자

## 배경
기술 리서치 결과는 `Node.js + Express + Socket.IO + PostgreSQL + Redis + Nginx` 조합을 권장하고 있다. 회의에서는 이 구조가 MVP와 운영 복잡도 기준으로 적절한지, 초기 범위를 더 줄일지 결정해야 한다.

## 핵심 질문
1. 초기 구조를 `단일 Node.js 앱 + PostgreSQL + Redis`로 고정할지?
2. 실시간 통신은 `Socket.IO`로 확정하고 SSE는 제외할지?
3. 멀티테넌트는 `/:slug/...` 경로 기반과 `tenant_id + RLS` 구조로 확정할지?

## 현황 분석
- 현재 시스템은 키오스크, 창구, 디스플레이의 양방향 실시간 동기화가 핵심이다.
- SSE는 읽기 전용 화면에는 적합할 수 있으나, 호출/재호출/관리 이벤트까지 포함하면 Socket.IO가 더 자연스럽다.
- 단일 DB + `tenant_id + RLS`는 SMB 다수 테넌트 운영에서 가장 단순하다.
- 초기엔 PM2 단일 프로세스로 시작하고, 이후 Redis adapter로 확장하는 단계적 접근이 가능하다.

## 작성자 초안 의견
- `단일 Node.js 앱 + PostgreSQL + Redis`로 시작하는 것이 맞다.
- 실시간은 `Socket.IO`로 확정하고, 상태 변경은 REST API에서 처리한 뒤 broadcast하는 구조를 유지해야 한다.
- 멀티테넌트는 `/:slug/...` 경로 기반과 `tenant_id + RLS`를 기본 구조로 확정하는 편이 안전하다.
- React/Next.js 여부는 기술 리서치보다 화면/개발 생산성 관점의 별도 결정으로 남기는 편이 낫다.
