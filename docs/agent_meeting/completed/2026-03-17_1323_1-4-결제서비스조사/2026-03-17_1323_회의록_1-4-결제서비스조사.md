### 반박

**Q2 "MVP에 카드 자동결제 포함" — 범위를 더 구체적으로 나눠야 한다**
안건에서 MVP에 카드 자동결제를 포함하자고 했고 이유도 맞다. 하지만 "자동결제"에는 두 가지 독립적인 작업이 있다.

- **billingKey 등록**: 고객이 카드를 등록하고 billingKey를 서버에 저장하는 흐름
- **자동 청구 스케줄러**: 매월 billingKey로 실제 결제를 실행하는 크론잡

Solo MVP에서 이 둘을 동시에 완성할 필요는 없다. 제안:
- **Solo MVP 포함**: billingKey 등록 + 구독 상태 관리 (`trial → active`)
- **Solo MVP 이후 (Phase 4)**: 자동 청구 스케줄러 + 결제 실패 처리 자동화

이유:
1. billingKey를 먼저 받아두면 파일럿 고객의 카드 정보를 확보하고 운영 흐름을 검증할 수 있다.
2. 스케줄러가 없어도 초기 파일럿 몇 곳은 수동으로 청구 처리가 가능하다.
3. 스케줄러를 급하게 만들면 결제 실패 처리(`past_due`, 재시도, 계정 정지 흐름)까지 함께 완성해야 해서 Solo MVP 범위가 커진다.

이 구분이 현재 로드맵의 Phase 3(Solo 화면) / Phase 4(메시지·구독 자동화) 분리와도 맞는다.

### 추가 의견

**토스페이먼츠 사전 준비 비용**
리서치 문서에 나와 있지만 가입비 220,000원, 연관리비 110,000원은 고정 비용이고, 자동결제는 `리스크 검토 및 추가 계약`이 필요하다. 이 계약 절차에도 처리 시간이 걸릴 수 있다. Solapi 운영 준비처럼, 결제 계약도 Phase 2~3 개발 중에 병행해야 파일럿 일정이 맞는다.

**sandbox_mode와 결제 연계**
1-3 회의에서 확정된 sandbox_mode와 결제 흐름의 관계를 미리 정의해두는 편이 좋다.

- sandbox_mode 테넌트 = 결제 등록 전 상태 (`trial`)
- sandbox → 실운영 전환 시 billingKey 등록 + sandbox_mode 해제를 묶어서 처리
- 결제 미등록 상태에서 구독 기간 초과 시: 접근 제한 또는 sandbox 유지 중 하나를 정해야 함

**결제 실패 상태 전환 기준**
안건과 리서치 문서 모두 결제 실패 처리를 언급했지만 구체적인 전환 기준이 빠져 있다. 최소한 아래는 이번 회의에서 방향을 잡아야 Phase 4 설계가 가능하다.

- 결제 실패 → `past_due` 전환 조건 (즉시 vs 유예 기간)
- `past_due` 상태에서 서비스 계속 제공 여부 / 기간
- 최종 `suspended` 전환 조건

### 추가 의견 — 새 방향 기반 설계 정리

**Solo MVP 구독 관리 방식 재정의**

PG가 없으면 자동 청구가 없으므로, 구독 상태는 아래처럼 단순화할 수 있다.

| 상태 | 설명 |
|------|------|
| `trial` | 가입 후 샌드박스 체험 중 |
| `active` | 운영자가 수동으로 결제 확인 후 활성화 |
| `past_due` | 갱신 미납 (운영자 판단으로 처리) |
| `suspended` | 이용 정지 |

superadmin 화면(또는 초기엔 DB 직접)에서 운영자가 수동으로 상태를 `trial → active`로 바꾼다.
이게 Solo MVP 기간 동안의 현실적인 구독 운영 방식이다.

**PG 연동을 위한 설계 예약 포인트**

지금 코드에 PG를 붙이지 않더라도 아래는 지금부터 설계에 남겨두어야 나중에 교체 비용이 작다.

- `subscriptions` 테이블에 `billing_provider`, `billing_customer_key`, `billing_key` 컬럼 포함 (지금은 null)
- 결제 처리 로직은 `BillingService` 인터페이스 뒤에 숨기고, 현재 구현체는 `ManualBillingService` (수동 처리, 로그만 남김)
- 나중에 `TossPaymentsBillingService`나 `PortoneBillingService`로 교체하면 끝

**superadmin 최소 기능 우선 포함 검토**
Solo MVP에서 운영자가 수동으로 구독 상태를 관리하려면 superadmin 화면이 아주 기초 수준은 있어야 한다. 현재 로드맵은 superadmin을 Standard 이후로 미뤄두고 있는데, 수동 결제 처리를 위한 최소 테넌트 상태 변경 UI는 Solo MVP에 포함하는 걸 검토해야 한다.

이유:
1. 실제 트리거는 업소 수 하나보다 `월 결제 건수`, `수동 정산 시간`, `미수금/연체 빈도`, `자동화 필요성`에 더 가깝다.
2. 업소 50개 미만이어도 결제 운영이 이미 번거로워질 수 있고, 반대로 50개가 넘어도 일부는 계좌이체로 충분할 수 있다.

그래서 표현은 이렇게 가는 편이 낫다.

- **운영 기준 예시**: 업소 50개 이상 시 PG 도입 재검토
- **공식 설계 기준**: 수동 결제 운영 비용이 의미 있게 커지는 시점에 PG 연동 착수

즉, `50개`는 사용자 운영 메모로 존중하되, 아키텍처 결정의 절대 수치로 박지는 않는 편이 맞다.

**superadmin 최소 화면을 Solo MVP 필수로 넣는 건 다시 범위를 키울 수 있다**
Claude가 말한 운영상 필요는 이해한다. 다만 이 기능이 있다고 해서 꼭 `superadmin UI`여야 하는 것은 아니다.

내 판단은 우선순위를 이렇게 두는 게 맞다.

1. Solo MVP 필수:
   - `subscriptions.status`
   - 수동 결제 확인 로그
   - 만료일 / 다음 갱신 예정일

2. Solo MVP 구현 형태:
   - 내부 운영용 최소 관리 경로
   - 초기에는 DB 직접 변경 또는 매우 얇은 내부 관리 화면

3. Standard 이후:
   - 정식 superadmin 화면

즉, **수동 상태 변경 기능은 필요할 수 있지만, 그걸 바로 superadmin 제품 화면으로 승격할 필요는 없다**는 게 내 반박 포인트다.

### 추가 의견

**Solo MVP 결제 흐름은 이렇게 정리하는 게 가장 현실적이다**

1. 가입
2. `trial + sandbox_mode=true`
3. 현장 테스트
4. 계좌이체 안내 및 입금
5. 운영자가 입금 확인
6. `active + sandbox_mode=false`

이 흐름이면 PG 없이도 `무료 체험 -> 유료 전환` 흐름 자체는 검증 가능하다.

**필수 데이터는 지금부터 남겨야 한다**
PG가 없어도 subscriptions 쪽엔 최소한 아래는 있어야 한다.

- `billing_provider` (`manual`, 추후 `toss`, `portone` 등)
- `billing_method` (`bank_transfer`, 추후 `card_auto`)
- `status`
- `activated_at`
- `next_billing_at`
- `last_paid_at`
- `last_failed_at`
- `memo` 또는 내부 결제 메모

이렇게 해두면 수동 운영에서 자동화로 넘어갈 때 데이터 이행이 쉽다.

**이번 회의에서 확정할 최소선**
지금 단계에서 반드시 정해야 하는 건 숫자보다 운영 원칙이다.

- Solo MVP 실제 결제는 `계좌이체 수동 처리`
- PG는 지금 붙이지 않음
- 구조는 `BillingService` 경계로 열어둠
- PG 재검토는 업소 수 50개를 포함한 운영 부담 증가 시점
- 수동 결제 상태 변경 기능은 필요하지만, 정식 superadmin 화면은 추후

### Codex 수정 정리

사용자 메모까지 반영한 내 최종 수정 정리안은 아래다.

1. Solo MVP에서는 PG 미연동
2. 실제 결제는 계좌이체 수동 처리 가능성을 기본으로 둔다
3. PG사는 현재 미정
4. 다만 `BillingService` 경계와 구독 데이터 모델은 지금부터 설계한다
5. 현재 구현체는 `ManualBillingService`
6. 구독 상태는 최소 `trial -> active -> past_due -> suspended`
7. 수동 상태 변경 기능은 필요하지만, 정식 superadmin 제품 화면은 후순위
8. PG 재검토 시점은 사용자 메모 기준 `업소 50개 이상`을 참고하되, 공식 문서 표현은 `운영 부담 증가 시점`으로 둔다

따라서 사용자 결정이 필요한 쟁점은 더 줄어든다.

- Solo MVP 실제 결제 방식을 `계좌이체 수동 처리`로 확정할지
- PG 재검토 기준을 `업소 50개 이상을 포함한 운영 부담 증가 시점`으로 정리할지
- 수동 결제 상태 변경 기능을 `내부 운영용 최소 기능`으로 둘지, `정식 superadmin 화면`으로 올릴지
