## Claude (12:21)

### 동의

**Q1 알림톡 우선 + SMS fallback 방향**
안건의 기본 방향에 동의한다. 알림톡이 단가(13원)와 정보성 메시지 적합성 양쪽에서 유리하다. 리서치 문서의 판단대로 현재 서비스 메시지 유형(순번발급/호출예정/호출)은 카카오 알림톡 정책과 충돌할 이유가 없다.

**Q2 브라우저 발송 구조는 폐기 대상**
`kiosk-sms/index.html`의 현재 구조는 운영용으로 쓰면 안 된다. API 키가 localStorage에 노출되고, 서명 로직이 임시 랜덤 문자열이고, PFID가 placeholder 상태다. 이 파일을 실제 운영에 투입하는 건 보안·안정성 양쪽에서 위험하다. 폐기 대상으로 확정하는 것이 맞다.

---

### 반박

**Q1 SMS fallback — "선택적" 범위를 더 좁혀야 한다**
안건에서 "선택적 SMS fallback"이라고 표현했는데, 이걸 기본 ON으로 두면 Solo MVP에서 원가 통제가 어렵다. 특히 소규모 현장 판매 업종(호떡집, 팝업스토어 등)은 메시지 단가 하나하나에 민감할 수 있다.

제안:
- Solo MVP에서는 **알림톡 only**가 기본 (fallback 없음)
- SMS fallback은 관리자 설정에서 명시적으로 켜야 작동하는 옵션
- fallback을 켤 경우 관리자 화면에 "SMS는 18원/건이 추가로 발생합니다" 안내 문구 표시

**Q3 MVP에 템플릿/PFID 관리 UI가 필요한가?**
안건에서 MVP 필수 항목으로 "템플릿/PFID 관리"를 넣었는데, UI까지 Solo MVP에 포함하면 개발 범위가 커진다. PFID와 템플릿 코드는 초기에 운영자가 직접 DB에 입력하거나, 관리자 설정 화면에서 단순 텍스트 입력 방식으로 저장하면 충분하다. 별도 관리 UI는 Standard 단계로 미뤄도 된다.

### 추가 의견

**사전 준비 기간 문제**
알림톡 운영에는 카카오 비즈니스 채널 개설 → Solapi 연동 → PFID 발급 → 템플릿 검수 단계가 있고 각 단계마다 1~3 영업일이 걸린다. 파일럿을 목표 시점에 시작하려면 개발 완료 전에 이 준비를 병행해야 한다. 로드맵에 "파일럿 2~3주 전 Solapi 운영 준비 시작" 항목을 추가하는 것을 제안한다.

**sandbox_mode와의 연계**
이미 2-2 아키텍처에서 sandbox_mode 설계가 확정됐다. `sandbox_mode = true`이면 Solapi 호출을 스킵하고 sandbox 로그만 남기는 구조다. 이 설계가 있으면 Solapi 실제 연동 전에도 Solo 화면 전체를 테스트할 수 있다. 발송 구조 개발 순서는 "서버 발송 기본 구조(sandbox 분기 포함) → 파일럿 전 실제 PFID 연결"이 적합하다.

**친구톡 종료 (2025-12-31)**
리서치 문서에 나온 카카오 친구톡 서비스 종료 건은 현재 MVP 설계에 영향이 없다. 지금 설계가 알림톡 기반이고 친구톡 의존 요소가 없으므로 이 항목은 단순 확인으로 닫으면 된다.

## Codex (12:50)

### 동의

**Q1 SMS fallback은 기본 OFF가 맞다**
Claude가 지적한 비용 통제 문제에 동의한다. 현재 확정된 Solo MVP 방향은 `소규모 현장 판매 업종 + 태블릿 1대`이고, 이 구간에서는 전달률 극대화보다 `원가 예측 가능성`이 더 중요하다. 따라서 운영 기본값은 `알림톡 only`, SMS fallback은 관리자 명시 활성화 옵션으로 두는 편이 맞다.

**Q2 브라우저 직접 발송 구조는 MVP 이전 폐기 대상이다**
이 부분도 동의한다. 이건 단순 개선 대상이 아니라 `운영 금지 대상`으로 분류해야 한다. 현재 `kiosk-sms/index.html` 구조를 남겨둔 채 서버 발송을 병행하면 운영 경로가 2개가 되고, 로그 정합성과 보안 기준이 동시에 무너진다. Solo MVP 기준 메시지 경로는 `클라이언트 -> 서버 API -> Solapi` 한 줄로 강제해야 한다.

**sandbox 기반 개발 순서**
`sandbox_mode`를 먼저 붙이고, 그 위에 실제 Solapi 연결을 올리는 순서도 동의한다. 이미 현재 로드맵도 Phase 2에서 `서버 발송 기본 구조 + sandbox 분기`, Solo MVP 컷라인에서 `Solapi 연결 시 기본 알림톡 발송 가능`으로 정리돼 있다. 이 순서가 구현/검증 비용이 가장 낮다.

### 반박

**Q3 "템플릿/PFID 관리 UI는 Standard 이후"로 밀면 너무 늦다**
Claude가 말한 취지는 맞다. Solo MVP에서 별도 복잡한 관리 콘솔까지 만들 필요는 없다. 다만 `운영자가 직접 DB 입력`을 기본 전제로 두는 건 MVP 기준으로도 과하다.

왜냐하면 현재 확정 로드맵상 Solo MVP 범위에는 이미 `/:slug/admin/messages` 기본 화면이 포함되어 있고, 이 화면은 최소한 아래 3가지는 처리해야 파일럿 운영이 가능하다.

- PFID 저장/수정
- 발신번호 저장/수정
- 템플릿 코드 또는 템플릿 식별자 입력/검증

즉, 반박 포인트는 이거다:
- **복잡한 관리 UI는 불필요**
- 하지만 **최소 입력 UI 자체는 Solo MVP에 필요**

이 선을 안 그으면 실제 파일럿 준비가 개발자 수작업에 묶여서 `셀프 설정 가능한 SaaS` 방향과 어긋난다.

**사전 준비 시점은 "파일럿 2~3주 전"보다 더 앞당겨야 한다**
알림톡 운영 준비를 파일럿 직전 작업처럼 두면 늦을 수 있다. PFID/템플릿 검수는 외부 리드타임이 있고, 실제 메시지 본문이 확정돼야 검수도 들어간다. 따라서 타이밍은:

- Phase 2에서 메시지 이벤트 종류 확정
- Phase 3에서 관리자 기본 입력 화면 확보
- 그 직후 운영 준비 병행

이 순서가 더 맞다. 즉, `파일럿 2~3주 전`이라는 단일 시점보다, `Phase 2~3부터 병행 시작`이라고 적는 편이 구현 현실과 맞다.

### 추가 의견

**fallback 발동 조건도 지금 좁혀야 한다**
`실패 시 SMS fallback`이라고만 쓰면 범위가 넓다. Solo MVP 이후 자동화 단계에서도 아래처럼 제한하는 편이 맞다.

- 템플릿 미등록, PFID 누락 같은 설정 오류: fallback 금지
- 알림톡 발송 실패 중 전달 채널 문제만 fallback 허용 검토
- fallback 사용 여부와 추가 비용은 관리자에 명확히 표시

설정 오류까지 SMS로 덮어버리면 운영자가 잘못된 설정을 계속 방치하게 된다.

**로그 최소 단위**
MVP에서도 `sms_logs`에는 최소한 아래는 남겨야 한다.

- message_type (`issue`, `ready`, `call`)
- channel (`ata`, `sms`)
- status (`previewed`, `queued`, `sent`, `failed`)
- sandbox 여부
- provider_status_code

이 정도는 있어야 관리자 기본 화면에서 실제 원인 파악이 가능하다.

### 사용자 의견 반영 — 알림톡 채널 구조 추가 검토

사용자가 중요한 구조적 질문을 제기했다. 알림톡 발송에 두 가지 경우가 있다.

**Case A — 플랫폼 공유 채널**
- SaaS 운영자가 단일 카카오 비즈니스 채널을 보유
- 모든 테넌트의 알림톡이 이 채널에서 발송됨
- PFID는 플랫폼 레벨에서 하나
- 수신자에게는 플랫폼 채널 이름으로 도착 (테넌트 브랜드 아님)

**Case B — 테넌트 개별 채널**
- 각 업체가 자신의 카카오 비즈니스 채널을 보유
- PFID를 각 업체가 `admin/messages`에 직접 입력
- 수신자에게는 해당 업체 이름으로 도착
- 초기 셋업 부담: 채널 개설 → Solapi 연동 → PFID 발급 → 템플릿 검수

---

### 추가 의견 — 두 케이스 모두 설계에 반영해야 한다

Solo MVP 1차 타겟(소규모 현장 판매 업종)을 고려하면, 어느 하나만 선택하는 것보다 **하이브리드 구조**가 현실적이다.

**이유:**
- 호떡집, 팝업스토어 같은 소규모 업체는 카카오 비즈니스 채널을 별도로 개설하기 어렵다. 플랫폼 공유 채널이 빠른 온보딩을 가능하게 한다.
- 반면 학원, 의원처럼 브랜드 인지가 중요한 2차 타겟 업체는 자기 채널에서 메시지가 나오길 원한다.

**제안 구조:**

| 채널 방식 | 설명 | Solo MVP 포함 여부 |
|-----------|------|--------------------|
| 플랫폼 공유 채널 | 플랫폼 PFID 기본 사용, 테넌트 셋업 불필요 | ✅ 기본 제공 |
| 테넌트 개별 채널 | 업체가 자체 PFID 입력, 브랜드 발신 | ✅ 선택적 설정 |

구현 방식:
- `tenant_settings`에 `solapi_pfid`, `solapi_sender` 컬럼 추가
- 값이 비어있으면 → 플랫폼 공유 채널 사용
- 값이 있으면 → 해당 테넌트 채널 사용
- `admin/messages` 화면에서 테넌트가 직접 입력/변경 가능

이 구조면 Solo MVP에서 추가 셋업 없이 바로 알림톡 체험 가능하고(플랫폼 채널), 이후 브랜드 발신으로 업그레이드도 가능하다.

**템플릿 처리:**
- 플랫폼 공유 채널: 플랫폼이 미리 등록한 템플릿 사용 (업체명·번호를 변수로 치환)
- 테넌트 개별 채널: 업체가 자체 템플릿 코드를 `admin/messages`에 입력

## Codex (13:11)

### 동의

**채널 구조를 지금 설계 질문으로 끌어올린 건 맞다**
Claude가 추가한 Q3는 실제로 중요한 질문이다. 이걸 지금 안 정하면 `admin/messages` 범위, PFID 저장 방식, 템플릿 관리 방식이 전부 흔들린다. 특히 현재 문서 흐름상 Solo MVP에도 `admin/messages` 기본 화면이 포함되어 있으므로, "무슨 값을 입력받는 화면인지"를 먼저 정해야 한다.

**데이터 모델은 하이브리드로 열어두는 게 맞다**
플랫폼 공유 채널과 테넌트 개별 채널을 모두 수용할 수 있게 설계하는 데 동의한다. 이건 나중에 되돌리기 쉬운 선택이 아니다. 지금부터 아래 정도는 열어두는 편이 맞다.

- `message_channel_mode` (`platform_shared`, `tenant_owned`)
- 테넌트 override용 PFID / 발신번호 / 템플릿 식별자
- 로그에 실제 발송에 사용된 채널 모드 기록

이 정도를 열어두면 Solo에서 빠르게 시작하고, Standard나 2차 타겟으로 갈 때 재작성 비용이 작다.

---

### 반박

**하지만 "플랫폼 공유 + 테넌트 개별 채널 둘 다 Solo MVP에서 동급 지원"은 과하다**
여기서는 Claude 의견을 조금 좁혀야 한다. 구조를 둘 다 수용하는 것과, 둘 다 MVP 운영 기본으로 지원하는 것은 다르다.

내 판단은 아래다.

- **설계**: 하이브리드로 간다
- **Solo MVP 기본 운영**: 플랫폼 공유 채널 우선
- **테넌트 개별 채널**: 고급 옵션 또는 파일럿 예외 적용

이유는 단순하다.

1. Solo MVP 1차 타겟은 소규모 현장 판매 업종이다. 이들은 온보딩 속도가 핵심이라서, 개별 카카오 채널 개설/연동/PFID 발급을 기본 요구로 두면 진입장벽이 너무 높다.
2. 테넌트 개별 채널은 기술적으로는 단순해 보여도 운영 복잡도가 크다. PFID만 받는다고 끝나지 않고, 템플릿이 그 채널에 실제로 검수 완료됐는지, 변수 형식이 맞는지, 호출/발급 이벤트별 코드가 다 있는지까지 검증해야 한다.
3. 이 검증을 MVP에서 완전 자동화하지 않으면, 결국 관리자 화면이 있어도 운영 지원이 수작업으로 남는다.

즉, **채널 구조를 하이브리드로 확정하는 것**에는 동의하지만, **Solo MVP 기본 제공 범위를 하이브리드 양쪽 동급 지원으로 해석하면 범위가 커진다**는 점은 분명히 해야 한다.

### 추가 의견

**권장 단계 구분**
가장 현실적인 단계 구분은 아래다.

1. **Solo MVP 기본**
   - 플랫폼 공유 채널 사용
   - 알림톡 only
   - `admin/messages`에서는 연결 상태, 로그, sandbox 기록, 고급 설정 진입만 제공

2. **Solo MVP 고급 옵션 / 파일럿 예외**
   - 테넌트 개별 PFID / 템플릿 식별자 입력 허용
   - 단, 자동 검증 완성 전까지는 "운영 승인 후 사용" 수준으로 제한

3. **Phase 4 운영형 완성**
   - 템플릿/PFID/발신번호 관리
   - 개별 채널 검증 흐름
   - fallback 및 과금 자동화

이렇게 나누면 현재 로드맵과도 맞는다. Phase 3의 `admin/messages`는 기본 화면이고, Phase 4가 운영형 완성이다. 따라서 테넌트 개별 채널의 본격 지원은 Phase 4에 두는 편이 문서 정합성도 좋다.

**`admin/messages` 범위도 여기 맞춰 다시 정리해야 한다**
앞에서 내가 Solo MVP에 `PFID/발신번호/템플릿 식별자 입력`이 필요하다고 말했는데, 이건 "테넌트 개별 채널을 Solo MVP에서 기본 지원한다면"이라는 전제가 붙는다.

지금 채널 구조를 다시 좁히면 Solo MVP 기본 화면 범위는 이렇게 정리하는 게 더 맞다.

- 필수:
  - 메시지 정책 표시 (`알림톡 only`, sandbox 여부)
  - 플랫폼 채널 사용 상태
  - 최근 발송 로그 / 실패 코드
  - 고급 설정 진입
- 선택:
  - 테넌트 개별 PFID / 템플릿 식별자 입력

즉, `admin/messages` 기본 화면은 유지하되, **플랫폼 공유 채널 기본안**에서는 PFID 입력 UI를 필수로 볼 필요는 없다.

### Codex 수정 정리

지금까지 논의를 반영한 내 수정 정리안은 아래다.

1. 공급자는 `Solapi` 유지
2. Solo MVP 기본 메시지 정책은 `알림톡 only`
3. SMS fallback은 `기본 OFF + 관리자 명시 활성화`
4. 브라우저 직접 발송 구조는 `MVP 이전 폐기`
5. 채널 구조는 **설계상 하이브리드**로 간다
6. 하지만 Solo MVP 기본 운영은 **플랫폼 공유 채널 우선**
7. 테넌트 개별 채널은 **고급 옵션/파일럿 예외**로 시작하고, 운영형 완성은 Phase 4에서 다룬다
8. `admin/messages` Solo MVP 기본 범위는 **정책 표시 + 로그 확인 + 고급 설정 진입** 중심

따라서 사용자 결정이 필요한 쟁점은 다시 아래 3개로 정리된다.

- Solo MVP 기본 메시지 정책을 `알림톡 only`로 확정할지
- 채널 구조를 `설계는 하이브리드, 기본 운영은 플랫폼 공유 채널 우선`으로 확정할지
- Solo MVP `admin/messages` 범위를 `기본 로그/정책 화면` 중심으로 둘지, `개별 PFID 입력`까지 기본 포함할지
