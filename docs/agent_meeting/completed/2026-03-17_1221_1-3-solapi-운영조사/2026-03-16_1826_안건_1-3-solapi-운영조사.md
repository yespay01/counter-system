# 안건 — 1-3 Solapi 운영 조사 (2026-03-16 18:26)

> 작성: Codex
> 요청자: 사용자

## 배경
메시지 공급자는 Solapi로 유지하기로 했고, 현재 브라우저 직접 발송 구현은 운영용으로 부적합하다는 결론이 정리됐다. 회의에서는 Solapi 운영 범위와 서버 이관 우선순위를 확정해야 한다.

## 핵심 질문
1. 기본 메시지 채널을 `알림톡 우선 + 선택적 SMS fallback`으로 확정할지?
2. 현재 `kiosk-sms/index.html`의 브라우저 발송 구조를 즉시 폐기 대상으로 볼지?
3. MVP 범위에 `발송 로그`, `실패 재시도`, `템플릿/PFID 관리`를 어디까지 포함할지?

## 현황 분석
- Solapi는 현재 제품 흐름과 잘 맞고, 대기 발급/호출 같은 정보성 메시지에 적합하다.
- 알림톡 운영에는 발신번호 등록, PFID, 템플릿 검수 등 사전 준비가 필요하다.
- 현재 코드에는 브라우저 `localStorage`에 키 저장, 임시 서명, placeholder PFID 등 운영상 위험 요소가 남아 있다.
- 서버 발송 구조로 옮기지 않으면 SaaS 운영 보안과 감사 로그를 확보하기 어렵다.

## 작성자 초안 의견
- 공급자는 `Solapi`로 유지한다.
- 메시지는 `알림톡 우선`, 실패 시 조건부 `SMS fallback`을 권장한다.
- 브라우저 직접 발송 구조는 MVP 이전 단계에서 정리해야 한다.
- MVP에는 최소한 `서버 발송`, `발송 로그`, `템플릿/PFID 관리`, `실패 상태 표시`까지는 포함해야 한다.
