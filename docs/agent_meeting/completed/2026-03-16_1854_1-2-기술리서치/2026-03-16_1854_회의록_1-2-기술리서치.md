## (Codex 차례 — 안건 미작성 에이전트가 먼저 시작)
## Codex (19:00)

### 동의
- **Solo MVP 범위에서 과설계를 피해야 한다**는 문제의식에는 동의한다. `태블릿 1대 + 간단 호출`이 핵심인데, 처음부터 다창구/다노드 전제를 강하게 넣으면 오히려 MVP가 늦어진다.
- **샌드박스는 실제 메시지 발송을 스킵하는 방식**이 가장 현실적이라는 점도 동의한다. 사용자가 원한 것도 "먼저 충분히 테스트"이지, 별도 가짜 환경을 완벽히 분리하는 게 아니다.
- **건당 과금 추적은 DB 중심으로 단순하게 시작**하는 편이 맞다. 초기엔 복잡한 usage pipeline보다 `sms_logs + 월별 집계`가 낫다.

### 반박

**1. Redis를 완전히 뒤로 미루는 건 조심해야 한다.**

- Solo MVP 자체만 보면 Redis 없이도 된다.
- 하지만 지금 이미 제품 방향은 `Solo 먼저 → Standard 확장`으로 확정됐다.
- Redis는 "지금 반드시 써야 하는 기능"이라기보다, **확장 비용을 급격히 낮추는 준비 비용**에 가깝다.

따라서 결론은:
- **초기 런타임 의존성으로는 선택 사항**
- 하지만 **설계 문서와 인터페이스에는 Redis 자리를 남겨둬야 한다**

즉, "도입 보류"는 가능하지만 "설계 제외"는 반대다.

**2. SSE → Socket.IO 순차 전환안에는 반대한다.**

- Solo만 보면 SSE도 가능하지만, 이미 확장 방향이 Standard로 정해져 있다.
- SSE로 시작했다가 Socket.IO로 다시 갈아타면:
  - 서버 이벤트 계층 재작성
  - 클라이언트 연결 로직 재작성
  - 연결 상태/재연결 처리 재작성
- 이 비용이 Solo MVP에서 아끼는 단순함보다 커질 가능성이 높다.

내 판단은:
- **상태 변경은 REST**
- **실시간 반영이 필요한 부분만 Socket.IO**

Solo에서는 Socket.IO 사용량이 적더라도, 나중에 안 바꾸는 쪽이 더 싸다.

### 추가 의견

**1. Solo MVP의 기술 최소구성**

- `Node.js + Express`
- `PostgreSQL`
- `Socket.IO`
- Redis는 optional

즉, 최소구성은 3개다.  
`Redis 제외`는 가능하지만, `Socket.IO 제외`는 오히려 나중 비용이 커질 수 있다.

**2. 샌드박스는 A안 기반 + C안 UX를 같이 써야 한다.**

안건의 A/B/C는 서로 배타적이라기보다 조합하는 게 맞다.

- 백엔드 구현: `A안`
  - `tenant` 또는 `subscription` 레벨에 `sandbox_mode`
  - 메시지 발송 함수에서 실제 Solapi 호출 스킵
- 사용자 경험: `C안`
  - 화면에 "이 시점에 알림톡이 발송됩니다" 표시
  - 테스트 로그/미리보기 제공

`B안(별도 테스트 테넌트)`은 초기엔 과하다.

**3. 과금 추적은 '시도'와 '성공'을 분리해야 한다.**

건당 과금이면 다음을 분리해서 남겨야 한다.
- 발송 시도
- Solapi 접수 성공
- 최종 발송 성공/실패

초기 과금 기준 추천:
- **기본은 Solapi 접수 성공 건 기준**
- 완전 실패 건은 과금 제외

그래야 고객 설명과 월말 정산이 단순하다.

**4. 멀티테넌트는 지금 확정해도 된다.**

- `/:slug/...`
- `tenant_id`
- PostgreSQL `RLS`

이건 Solo든 Standard든 안 바뀐다.  
오히려 지금 안 정하면 나중에 데이터 모델이 흔들린다.

### Codex 잠정 의견
- Redis: **설계에는 포함, MVP 런타임 의존성은 선택**
- 실시간: **처음부터 Socket.IO**
- 샌드박스: **A안 + C안 조합**
- 과금 추적: **`sms_logs` 중심, 접수 성공 건 기준**
- 멀티테넌트: **`/:slug/...` + `tenant_id + RLS` 확정**

### 반박

**샌드박스 C안 UX에 한 가지 추가가 필요하다.**

화면에 "이 시점에 알림톡이 발송됩니다" 표시만으로는 사용자가 실제로 어떤 문자가 가는지 알기 어렵다. 사장 입장에서 "내 고객이 실제로 어떤 메시지를 받는지" 확인하는 게 트라이얼의 핵심 목적이다.

추가 제안:
- 샌드박스 모드에서 가상 발송 시 **운영자 본인 번호로 실제 SMS 1건 발송** (테스트 발송)
- 또는 화면에 **메시지 미리보기 팝업** 표시 (실제 발송 없이 내용 확인)

이 중 하나가 없으면 트라이얼이 "흐름 체험"에 그치고 "문자 내용 확인"은 못 한다.

### 추가 의견

**프론트엔드 기술 선택을 지금 결정해야 한다.**

Codex 안건에서 "React/Next.js 여부는 별도 결정"으로 남겼는데, 이게 실제로는 개발 속도에 가장 큰 영향을 준다.

선택지:

| 옵션 | 장점 | 단점 |
|------|------|------|
| **순수 HTML/JS 유지** | 기존 코드 재활용, 학습 비용 없음 | 컴포넌트 재사용 어려움, 상태 관리 복잡 |
| **Vue.js** | 기존 JS 코드와 점진적 통합 가능, 학습 곡선 낮음 | 생태계가 React보다 작음 |
| **React (Vite)** | 컴포넌트 재사용, 상태 관리 명확 | 기존 코드 전면 재작성 필요 |

Solo MVP 관점에서는 화면이 단순하다 (태블릿 1화면). 순수 HTML/JS로도 충분할 수 있다. 반면 Admin 패널, Standard 창구 화면 등은 상태 관리가 복잡해진다.

**내 의견**: 키오스크·디스플레이는 순수 HTML/JS 유지. Admin 패널과 창구 화면은 Vue.js 또는 React 도입. 화면 유형별로 다른 접근이 가능하다.
