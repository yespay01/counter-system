## 배경

1-1 시장조사 회의에서 아래 사항이 확정됐다:
- **MVP**: Solo 모드 (태블릿 1대, 키오스크+호출 통합)
- **1차 타겟**: 소규모 현장 판매 업종 (호떡집, 분식 포장점, 팝업스토어 등)
- **가격**: 기본 구독료 + 메시지 건당 별도 과금
- **트라이얼**: 샌드박스 방식 (실제 SMS 발송 없이 전체 흐름 체험)

이 결정들이 기술 스택 선택에 영향을 준다. 특히 Solo 모드 MVP라는 점에서 초기 구조를 얼마나 단순하게 가져갈지, 그리고 샌드박스 트라이얼을 어떻게 구현할지가 핵심 기술 질문이 됐다.

---

## 핵심 질문

1. **Solo MVP 기준으로 Redis가 초기부터 필요한가?**
   - Solo 모드는 태블릿 1대 단일 사용자. 실시간 동기화 복잡도가 낮다.
   - Redis 없이 `Node.js + PostgreSQL`만으로 MVP를 시작하고, 다창구 필요 시 추가하는 방안 검토 필요.

2. **실시간 통신: Socket.IO vs SSE**
   - Solo 모드는 키오스크→서버→사장 폰 단방향에 가깝다.
   - SSE(단방향)로 Solo MVP를 커버하고, Standard(다창구) 단계에서 Socket.IO로 전환하는 방안이 현실적인가?

3. **샌드박스 트라이얼 구현 방식**
   - 실제 SMS를 발송하지 않고 전체 흐름(번호 발급→호출→알림)을 체험하게 하는 방법:
     - A. DB에 `is_sandbox` 플래그를 두고 SMS 발송만 스킵
     - B. 별도 테스트 테넌트 환경 제공
     - C. SMS 대신 화면에 "이 시점에 알림톡이 발송됩니다" 시뮬레이션 표시
   - 어느 방식이 구현 비용 대비 효과적인가?

4. **메시지 과금 추적 구조**
   - 기본료 + 건당 과금 방식이므로 발송 건수를 테넌트별로 정확히 집계해야 한다.
   - SMS 발송 → 성공 응답 → DB 기록 → 월말 청구 흐름을 어떻게 설계할지.
   - Solapi 발송 실패 시 카운트 처리 방법.

5. **멀티테넌트 격리 방식 확정**
   - `/:slug/...` 경로 기반 + `tenant_id + PostgreSQL RLS` 구조로 확정할지.

## 현황 분석

- 기존 코드: 순수 HTML/JS + Firebase. 백엔드 없음, 서버 사이드 로직 없음.
- Solo MVP는 창구 배정·디스플레이 없이 `번호 발급 → 대기 → 호출 → SMS` 4단계만 필요.
- 이 4단계는 실시간 동기화 요구가 낮다. 사장이 버튼 누르면 고객 폰으로 SMS 가는 구조.
- 기술 리서치 문서(`1-2-technical-research.md`)는 풀스택 구조를 권장하고 있으나, Solo MVP 범위에서는 과한 부분이 있을 수 있다.

---

## 작성자 초안 의견

- Solo MVP는 `Node.js + Express + PostgreSQL` 3개로 시작할 수 있다. Redis는 Standard 단계에서 추가.
- 실시간은 SSE로 Solo를 커버하고, Standard에서 Socket.IO 도입. 전환 비용이 크지 않다.
- 샌드박스는 A안(`is_sandbox` 플래그)이 가장 구현 비용이 낮다. DB 구조 변경 없이 SMS 발송 분기만 추가.
- 메시지 과금은 `sms_logs` 테이블 + 월별 집계 쿼리로 충분. 별도 과금 서비스 불필요.
